# Feishu Notifier 设计原则

> 这些原则基于用户实际使用反馈，请在开发新功能或修改现有功能时遵循。

## 核心使用场景

**主动查看状态** - 用户最常使用的场景

当用户发送任意消息（非命令）时，展示 OpenCode 当前状态。

## 设计原则

### ✅ 要做的

1. **信息简洁、有价值**
   - 进度百分比（如 `████░░░░░░ 40%`）
   - 当前动作（正在执行什么）
   - 已完成的关键步骤
   - 避免冗余信息，直击重点

2. **所有操作都通过 trigger 按钮**
   - 用户不想手动输入命令
   - 点击按钮即可完成操作
   - 减少用户输入成本

3. **每次查看都是新状态快照**
   - 用户想看时就发送一条新消息
   - 显示发送时刻的状态
   - 状态是即时获取的

### ❌ 不要做的

1. **不要卡片更新功能**
   - 用户认为"这个功能很蠢"
   - 不自动更新已发送的卡片
   - 每次查看都发送新消息

2. **不要批量操作**
   - 不需要"继续所有空闲会话"之类的批量按钮
   - 用户倾向于逐个处理

3. **不要信息过载**
   - 标题可能与实际内容不符（会话过程中内容会变）
   - 显示太多信息会让用户困惑
   - 只展示最有价值的信息

## 状态显示格式

```
📋 正在执行

进度 ████████░░ 80% (4/5)
当前动作 ▶️ 实现用户验证逻辑
已完成
  ✅ 分析现有代码结构
  ✅ 设计重构方案
  ✅ 实现核心模块

💡 回复「继续」或点击按钮查看详情
```
